# superglyphs

Working area for the **superglyph** program: lifting the fixed glyph catalog and
its home-local translation rule from the 3×3 glyph up to the 255×255 block
inside a priority-≥8 cage.

This README is the shared-vocabulary reference. Definitions are ordered from the
ground up — each builds on the previous.

> **⚠️ The universe is SEEDED, not hand-rolled. (This has tripped up every
> session — read it.)** To get the symmetric four-quadrant universe — or any
> coherent finite map — **create the seed with `Propagation.fromUniverseExtents`
> (or `fromUniverseBoundary(Universe.create(…))`) and grow it by substitution.**
> Do **not** try to reconstruct the off-SE quadrants as reflections / D4-duals of
> the SE map: the four quadrants sit at the Home Anchor's four quads with
> *different N/W padding*, so naive reflection fails (tested — 0 matches). Seeded from a real
> `fromUniverseExtents` grid, the translation table expands **all four quadrants
> correctly** with no extra work. Verified: `expand(seed @ ext 32)` equals
> `universe @ ext 64` with **0 mismatches**, and every seed glyph already has a
> rule in `TRANSLATION_V`. The universe is a substitution fixed point *about its
> centre* — so descent/expansion from the seed reaches all four quadrants. When in
> doubt: **seed it, don't derive it.**

## Coylean map

The object this whole repository studies. It is a deterministic field of
arrows on the 2-D dyadic integer grid, generated by **one local rule tiled
everywhere** — there is no global formula, only the rule and a boundary feed.

Each cell:

- takes **two boolean inputs** — one from the north edge (a "down" arrow `d`
  entering from above), one from the west edge (a "right" arrow `r` entering
  from the left);
- emits **two boolean outputs** — south (`d`) and east (`r`);
- combines them by an **XOR rule**, with the tie (both inputs active) broken by
  the cell's **priority orientation**, V (vertical wins) or H (horizontal wins).

Propagation flows **south-east**. The priority orientation of each cell is fixed
by the **2-adic valuation of its absolute index** (`pri(n)`), which is what makes
the map self-similar under dyadic dilation. A finite, coherent map is obtained
from `Propagation.fromUniverseBoundary(Universe.create({...}))` — never by
hand-seeding a single arrow.

Everything below is a structure *inside* this one map.

## Reaction box

The **atom** of the map: a single cell viewed as a machine. 2-in / 2-out, **one
boolean per port** (N-in, W-in → S-out, E-out), plus a priority orientation.

It is **invertible**: given any two of the four ports plus the priority, the
other two are determined, and the box is symmetric under interchanging
complementary inputs and outputs (N-in↔S-out, W-in↔E-out). The map *is* this box
tiled across the dyadic grid; every larger fact should be derivable from box-level
properties. (Higher-dimensional reaction boxes — the 3-cube and tesseract priority
CNOT cascades — live in `meta/4d/src/`.)

## Scaffold

The **representation** of the (infinite) map as nested **seams** rather than a
dense raster. Because propagation is SE-flowing, a K×K block is fully determined
by its north seam and west seam; the big-map engine stores only block-boundary
seams and reconstructs any tile on demand. The scaffold **grows N and W** by
concatenating the NE/SW/NW quadrant integrations onto the SE one in the same
seam format — the mathematically consistent way to extend a finite window is
`fromUniverseBoundary`, *not* overlaying four independent quadrants (they
disagree at the shared origin row/column).

See `meta/big-map/` (engine) and `meta/big-map/NOTES_lazy_scaffold.md` (the lazy
per-quadrant + rAF design).

## Glyph

An **order-1 block**: a **3×3 array of reaction boxes** (`NUM_CELLS = 3`). Each
of its four seams carries **3 cells**, not a single boolean.

- **Inputs:** a 3-bit north code + a 3-bit west code → `3 + 3 = 6` input bits →
  **64 glyphs**. The south and east seams (3 bits each) are *outputs*, fixed by
  propagating the 3×3 interior.
- The 64 glyphs are a **fixed catalog**, canonical at the `010` ("middle-of-3
  senior") priority pattern, **independent of dyadic location**. Moving the
  dyadic offset moves *which glyph lands in which map cell*; it never changes a
  glyph's own definition.

Lives in `glyphs/` (`glyph-core.js` is the Node-importable math layer).

## The Home Anchor (and its quads)

Every offset value is an **anchor** — a node of the dyadic quadtree carrying
four quadrant children (NW/NE/SW/SE). One is distinguished: the **Home Anchor**,
the origin and the dilation fixed point. In raw coordinates (`raw = offset − 1`)
the doubling map `x ↦ 2x` fixes only `x = 0`. (We say "away from home," not
"off-anchor" — there is no off-anchor; everything is an anchor and one of them
is home.)

The Home Anchor's **four quads** are the **dyadic offsets with latitude,
longitude ∈ {0, 1}**: `0/0, 0/1, 1/0, 1/1` (latitude = `vInitRow`, N–S; longitude
= `hInitCol`, E–W) — the four corners of the unit dyadic cell at the origin, and
the four quadrants of the canonical universe (the same canonical map with
different N/W padding).

Their significance is algebraic, not merely positional. The glyph **translation
rule** — the 4→1 dilation that sends a parent cage at absolute lattice
`(Pr, Pc)` to its four children at `(2Pr−1 + 4i, 2Pc−1 + 4j)`, `i,j ∈ {0,1}` — is
an **exact fixed point, a function of the glyph code alone, only on the Home
Anchor's quad family** `{0,1}²`. (Translation is the full 2×2 step; it factors
into two 2→1 substitutions — see *Genealogy of a superglyph*.) There exactly
**35 codes = 12 D4 orbits** appear, and that set *is* the whole table (for both V
and H).

Anywhere beyond the home quads (either coordinate ≥ 2) an offset drifts `×2` per
cage-level away from home, and the rule is no longer a function of the code: the
same code expands to different children in different places. It becomes a clean
function only of the larger **(self, North, West) tromino**, and that table is
**offset-specific**. So the home quads are the **only offsets where "glyph
translation" works** — where growth is a code→codes rule rather than a
position-dependent neighborhood rule.

## Cage

A region **fenced by high-priority lines**. Cells whose absolute index has 2-adic
valuation **≥ k** — positions divisible by `2^k` — form the **priority-≥k spine**.
Between two adjacent spine lines sits an interior of `2^k − 1` cells per axis.

Because **seniority decides the frame before the interior**, a cage boundary is a
closed seam and the interior is the substitution-closure of that seam. Cages nest:
the dyadic grid is a hierarchy of cages, every priority-≥(k+1) cage containing
priority-≥k cages.

| priority-≥k cage | interior per axis | this project calls it |
|---|---|---|
| k = 2 | `2²−1 = 3`   | **glyph** |
| k = 8 | `2⁸−1 = 255` | **superglyph** |

(The catalog/diamond pages count cells *inclusive* of both walls — `2^n + 1` per
axis — which is the same boundary, the `2^k − 1` interior plus the two shared
spine cells.)

## Superglyph

The **interior of a higher cage** — the generalization of a glyph one or more
dilation levels up. The case driving this directory is the **priority-≥8 cage**:
a **255×255 (`2⁸−1`) block** of reaction boxes, walled by valuation-≥8 lines.

Just as the 3×3 glyph is the unit at cage level 2, the 255×255 superglyph is the
unit at cage level 8. The research question is how the glyph's structure lifts:

- **Translation / substitution.** For glyphs, translation is home-local — clean
  only at the Home Anchor's four `{0,1}²` quads, with a 12-orbit alphabet. The
  open question is the superglyph analogue: a priority-≥8 cage nests a *sequence*
  of dilations, and the composite stays describable by translation only when
  every level lands its sub-offset back on the home quads. The admissible 255×255
  superglyphs
  should be the substitution-closure of the 12-orbit alphabet under the cage
  depth — a **subset** of all order-7-ish windows, not the full count. (Open;
  to be measured in Node against the existing `glyphs/test-tromino-crossorder`,
  `test-offset-regimes`, and abs-dilation checkers.)
- **Catalog.** Whether the superglyphs reachable inside a priority-≥8 cage form a
  fixed finite catalog the way the 64 glyphs do, and how many survive to that
  depth.

## Genealogy of a superglyph

The **genealogy** of a superglyph is its ancestry across cage levels. The two
directions have names:

- **Deconstruction** — finding a superglyph's **ancestors** (reducing toward the
  origin / coarser-to-finer-source).
- **Construction** — assembling **larger maps** from a superglyph (building
  outward / upward a cage level).

> **TODO — clever genealogical analogies.** This family-tree structure wants
> richer naming. The lead candidate, because it is *literal*: the **V
> substitution and the H substitution are the two parents** (mother / father),
> and the **translation is their child** — two single-axis lineages combining to
> produce one offspring, exactly like two parents creating a child. To develop
> from there: **inheritance** for which traits (codes/orbits) pass to the child,
> **DNA** for the seam/seed data that fully encodes a glyph's lineage, and
> whether **mitosis** (a single split) better names one substitution while
> **sexual reproduction** (two parents → child) names the V+H pair. Keep the
> analogies faithful to the math, not just cute.

Between scales there are **two reduction maps**, two paths of different grain:

- **Translation — 4 → 1.** The full isotropic dilation by 2: a `2×2` arrangement
  of four child cages collapses to one parent cage (the `(2P−1 + 4i, 2P−1 + 4j)`
  rule of the home quads, read in the ancestry direction). One translation =
  **one whole cage level** (`k → k+1`). This is the primary path for the
  superglyph program.

- **Substitution — 1 → 2** (construction) **/ 2 → 1** (deconstruction), and it
  **flips seniority** (V↔H). Two kinds:
  - **V substitution (v→h):** a `Vxx` glyph splits **horizontally** into a left
    and right **`Hxx`** glyph (with a bar between).
  - **H substitution (h→v):** an `Hxx` glyph splits **vertically** into a top and
    bottom **`Vxx`** glyph (with a bar between).

  A substitution moves only **half a cage level** and changes type V↔H, so it is
  the *square root* of translation. The intermediate "involving h" is literally
  the **H layer** sitting between two V layers. (These rules are the V→H / H→V
  cards on `glyphs/index.html`.)

**Two substitutions make a translation — v→h then h→v.** In construction: a V
glyph splits to a left/right **H** pair (v→h); each H glyph splits to a
top/bottom **V** pair (h→v); the result is four V glyphs — exactly the
translation's 2×2 (bars included). The middle layer is H; the round trip lands
back in V, one whole cage level deeper.

```
          v→h sub                       h→v sub
   1 V  ───────────→  H_L  H_R  ───────────────→   NW  NE     =  translation
        (horizontal                (each H splits     SW  SE      (back in V,
         split → 2 H)               vertically → 2 V)             one cage level)
```

**Verified** — `tests/two-substitutions-make-a-translation.mjs`. On the Home
Anchor's fixed-point map, v→h and h→v are each a clean function of the glyph code (0
conflicts; 27 V codes, 32 H codes), and composing the two rule tables reproduces
the direct 4→1 translation exactly (0 mismatches). So the **translation table is
the generalized shortcut** that collapses this V→H→V round trip into one
code→2×2 rule — it works on both V and H inputs because each is one leg of the
round trip.

**The half-step is a whole map.** Aggregated over an entire map, v→h turns a
whole V-order map into a real **H-order Coylean map of sideways glyphs**, wedged
dimensionally between its V neighbours:

```
  V_n      2ⁿ   × 2ⁿ     V glyphs (square)
  H_n      2ⁿ   × 2ⁿ⁺¹   H glyphs (sideways)   ← in between; v→h doubles columns
  V_{n+1}  2ⁿ⁺¹ × 2ⁿ⁺¹   V glyphs (square)     ← h→v then doubles rows
```

So the orders **interleave at half-cage grain** — V, H, V, H, … — and every H
map is a genuine intermediate (sideways) order, not bookkeeping. **Verified**
(`tests/between-every-two-v-orders-an-h-map.mjs`): the half-step rules learned at
one order regenerate the in-between H map *and* the next V map at other orders
(0 mismatches; alphabet 34 V / 35 H codes), so this is a transferable
substitution system, not a per-order coincidence.

**Bars too — resolved** (`tests/bars.mjs`). The 2×2's four separators are
recovered exactly from the half-steps (35/35): the single v→h bar IS both
vertical separators (`vTop = vBot = v→h bar`, unbroken — 35/35), and the two h→v
bars are the two horizontal separators (`hLeft`, `hRight`, independent). So a
superglyph's full "DNA" — codes *and* bars — descends through the V→H→V round
trip. (See *Bars are cage walls* below for what the bars then reveal about the
Home Anchor.)

> **Out of scope here — the (self, N, W) tromino.** The tromino rule is the
> machinery for the **away-from-home** regime; it does not belong to this
> Home-Anchor genealogy. Set it aside — superglyph genealogy is studied on the
> home quads only, where translation is a clean function of the code.

## Scaling up to the superglyph — the catalog closes

The rules are encapsulated as plain dictionaries in **`tests/rules.mjs`**
(`TRANSLATION_V/H`, `V_TO_H`, `H_TO_V`, each keyed by a glyph code `"d,r"`;
`node tests/rules.mjs` prints them readably, bars and all) plus `expandTranslation`
— a pure table lookup that doubles a map one cage level with no propagation.

**`tests/scale-up.mjs`** rides that from a small seed up to the superglyph scale
(priority-≥8 cage, order 8, 256 cells) and beyond, matching true propagation at
every order:

```
  order  sections  cells   codes  orbits   vs-truth
      5        8      32     27      12      0 ✓
      6       16      64     34      12      0 ✓
      7       32     128     35      12      0 ✓
      8       64     256     35      12      0 ✓   ← superglyph
      9      128     512     35      12      0 ✓
     10      256    1024     35      12      (skipped)
```

Two results:

- **The catalog closes.** The reachable alphabet saturates at **35 codes / 12 D4
  orbits** and never grows again. So a superglyph — and every larger cage — is
  tiled from the *same finite 12-orbit alphabet*; scaling up introduces no new
  glyphs. (This answers the "how many inner maps" question: finitely many, the
  closed set.)
- **A rule belongs to a code only once it has been a *parent*.** The 35th code
  first appears at order 7, so the table must ingest one level past that (7→8) to
  be closed; stop too early and scaling past order 7 hits ruleless cells. The
  table build in `rules.mjs` ingests 5→6→7→8 for exactly this reason.

Using the translation table (the 4→1 shortcut) is what makes this cheap:
each level is a doubling by lookup, not a fresh `O(4ⁿ)` propagation.

## Bars are cage walls — the Home Anchor's magic

A **bar** (separator) is a **cage wall**: a high-valuation priority line
dividing a cage into children. The bars turn out to *carry* the whole Home
Anchor property. Measured (`tests/bars.mjs`, order-7 parents = 961 cages):

| offset | broken vertical walls | wall-ambiguous codes |
|---|---|---|
| **1/1 Home Anchor** | **0** | **0** |
| 2/2, 5/3, 3/5 (away) | 174 | 27 / 46 / 58 |

At the Home Anchor every cage's vertical wall is **unbroken** (`vTop=vBot`)
*and* a function of the glyph code. Away from home the walls **break** and the
same code walls both ways.

**Proposed why** (ties the bars to the [[project_reaction_box]] /
conservation-of-information picture): a cage wall lies on a valuation line. At
the Home Anchor the priority lattice is aligned to the dyadic origin, so each
cage's midline falls on a line running the cage's **full height** → an intact
wall. An intact wall **seals** the cage (seniority resolves the wall before the
interior), so the interior is fed only through the cage's own seam → its
refinement is a pure function of its code = clean substitution = translation
works. Away from home the lattice is shifted (`firstDark ≠ 0`), the midline
straddles, the wall **leaks**, and refinement starts depending on neighbours —
the `(self, N, W)` tromino regime. In one line:

> **intact wall  ⟺  sealed cage  ⟺  code-determined substitution  ⟺  Home Anchor.**

## Saving work in big-map (home quads only)

Because the home map is a substitution fixed point, a section's glyph is a
function of its **dyadic address** — so you can compute the glyph at *any*
`(row, col)` of an order-M map by descending the translation table from a
one-section root: **O(M) lookups, touching no neighbours, propagating nothing**.

That's the saving for `meta/big-map`. The lazy scaffold builds blocks in an
SE-march, so block `(k1,k2)` waits on `(k1−1,k2)` and `(k1,k2−1)` — panning to a
far region drags in the whole upstream seam chain
(`NOTES_lazy_scaffold.md`'s `nextReadyAncestor` O(L) walk). Substitution replaces
that with **random access**: jump straight to the visible tile, seeded from one
tiny root, no chain and no propagation. A section glyph expands to its 4×4 cell
arrows via the catalog's per-glyph render, so full cell detail is kept.

**Verified** (`tests/random-access-tile.mjs`): address descent reproduces every
section of the order-8 map (0 wrong), spot-checks deeper orders against truth,
and pulls a single tile out of an order-30 map (~10⁹ cells/side — hopeless to
propagate) in 28 lookups / a few µs.

**The hard boundary is exactly the Home Anchor's quad family** `lat/long ∈ {0,1}`.
Away from home the cage walls break (above), translation is no longer a function
of the code, and big-map must fall back to propagation (or the offset-specific
tromino rule). The four quadrants of the canonical universe all sit at the Home
Anchor's quads (NW 0/0, NE 1/0, SW 0/1, SE 1/1), so the **default view qualifies
wholesale**; a user-dialled offset away from home does not.

**Live page — `universe.html`.** A scrollable map built on exactly this:
drag/zoom and jump to any cell up to `2⁴⁰` (≈1.1 trillion) per axis, each visible
section found by O(order) address descent (no propagation). Level-of-detail draws
real glyph arrows when zoomed in and coarse parent-glyph swatches when zoomed out.
The **cage walls** (the bars between sections) are address-determined the same way
— drawn on top, brighter + thicker for more outer cages (by the wall's 2-adic
valuation), so the nested cage hierarchy reads at any zoom.
It is the **full four-quadrant symmetric universe**, origin at the centre: built
the right way (see the ⚠️ note up top) — a `fromUniverseExtents` **seed**, sectioned
into glyphs, then descended; the seed carries all four quadrants and the table
expands them, so no reflections are derived. The page **"fakes the address"** —
underneath it is a 0-based propagation grid `[0, 2^d)`, but the user is shown
centred, signed coordinates so it reads as a universe with the origin in the
middle, not a propagation. Engine verified in Node (seed descent **and** walls ==
`fromUniverseExtents` at depth, 0 mismatches; address bits exact to `2³⁸`); the
live canvas is wired but unrun here.

**The broken-wall count is 2-adic, not a constant** (`tests/broken-wall-structure.mjs`).
The "174" that three offsets shared (2/2, 5/3, 3/5) was *not* an invariant —
they just fell in one dyadic band-cell. The count is a **self-similar function
of the offset's 2-adic address**:

- **Latitude gates** whether any wall breaks. A Cantor-like set of latitudes —
  size exactly `2^(order−2)`, doubling each order — is "active"; the rest
  (including {0,1}) are clean (0 broken). Rows within the active band are
  identical.
- **Longitude sets the magnitude** inside the active band, itself dyadically
  banded (e.g. at order 6: long 0→99, 1→130, {2–5}→39, {6–8}→120).

So the break-locus is a self-similar set (≈ ×4 per order at ~constant density) —
exactly what you'd expect from a substitution tiling — and there is **no constant
closed form**, only the address function. (`174` was an artifact of three offsets
in the same band-cell.)
